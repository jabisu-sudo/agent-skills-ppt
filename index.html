The Problem

🧠 AI Forgets Everything

  • Every session starts fresh — zero memory
  • "Use pnpm, not npm" → forgotten next day
  • "Follow conventional commits" → ignored
  • It's an architecture problem, not a bug

📄 Old Solutions Fall Short

  • Manual repeat — tedious, wastes context
  • Giant system prompt — burns 10K tokens, less room for actual work
  • RAG — heavy, retrieval quality inconsistent
  • Fine-tuning — expensive, slow to update

⚡ Layer 2: On-Demand Load

# User: "What's the weather in Suzhou?"
# → AI matches "weather" skill
# → Reads full SKILL.md into context:

## Weather Skill
### Steps
1. Parse city name from user input
2. Call: curl -s "wttr.in/{city}?format=j1"
3. Format: temp, humidity, wind, forecast
4. Fallback: Open-Meteo API if wttr.in fails

### Output Format
- Current: 🌡️ {temp}°C, {condition}
- Tomorrow: {high}/{low}°C

🗑️ Layer 3: Unload

Task done → SKILL.md dropped from context.
Next task loads a different skill.

Description = Routing Signal

"Not documentation for humans — a routing signal for AI."

❌ Bad

"A weather tool"

No trigger, no boundary → random firing

❌ Bad

"Helps you manage GitHub stuff"

2nd person, vague scope → conflicts with other tools

❌ Bad

"This skill sends messages to people via iMessage on macOS using AppleScript and the Messages app"

Implementation details, no trigger conditions

✅ Good

"Get current weather and forecasts via wttr.in.
Use when: user asks about weather, temperature, or forecasts.
NOT for: historical data, severe weather alerts."

✅ Good

"GitHub operations via gh CLI: issues, PRs, CI runs.
Use when: checking PR status, creating issues, viewing logs.
NOT for: complex web UI flows, bulk cross-repo ops."

3 rules: Use 3rd person · State trigger conditions · Define boundaries with "NOT for"

Design Choices

🎚️ Flexibility Spectrum

  • 🧱 Narrow Bridge — exact commands, no deviation
    curl -s "wttr.in/Suzhou?format=3"
  • 🌄 Open Field — goals + options, AI picks
    "Use wttr.in or Open-Meteo, choose best format"
  • Rule of thumb: AI keeps getting it wrong → add constraints; too rigid → loosen up

🧪 3 Failure Modes

  • Under-trigger: "苏州明天热不热" → skill not invoked
    → Add more keywords to description
  • Over-trigger: "I feel cold" → weather skill fires
    → Add "NOT for" boundaries
  • Exec error: Triggered correctly but API fails
    → Fix instructions in SKILL.md
